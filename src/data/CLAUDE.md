# src/data/ — Content

Static content: the plays themselves, the glossary, and roster name pools. No logic beyond the small index helpers.

## `plays/`
- **`*.ts`** — one file per published play, each exporting a single `Play` const (e.g. `flood.ts` → `export const flood`). These are **generated by the Designer's Publish flow** (`POST /api/designer/publish`, via `ts-morph`) but are plain TypeScript you can also hand-edit. They follow the flat step model and the branch-contiguity convention (see `src/types/CLAUDE.md`).
- **`index.ts`** — the registry. Imports every play into `ALL_PLAYS`, derives `PLAYS` (id → play lookup used by the `/plays/[playId]` route), and exposes the browse helpers `categoriesWithPlays()`, `setsInCategory(category)`, `playsInSet(category, set)`. Publishing a **new** play auto-adds its import + `ALL_PLAYS` entry here (ts-morph). Removing a play means editing both its file and this index.

To add a play, prefer the Designer → Publish path (dev only). Editing narrative can be done inline in the Viewer (dev only) or by hand in the play file — commit the result either way.

## `glossary.ts`
`GLOSSARY: Record<string, GlossaryEntry>` — ultimate-frisbee jargon (`break side`, `under`, `reset`, …). Each entry has a `definition` and optional `zone` (normalized rect). The Viewer's `NarrativeWithTooltips` underlines these terms in narrative and, on hover, shows the definition and highlights the zone on the field. Add a term here to make it tooltip-enabled everywhere.

## `names.ts`
Four arrays of real team names split by role and gender (`MMP/FMP` × `CUTTER/HANDLER`). `useRoster` samples these to assign display names to positions on each Viewer load. Purely cosmetic — positions (`C1`…`H3`) remain the real identifiers throughout the data model.
