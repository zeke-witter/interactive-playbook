# docs/ — Project documentation

Prose context and history. Not code — but the best source of *why* the app is shaped the way it is.

- **`design/design-standards.md`** — the most current product/UX briefing: what the app is, its two audiences (Designer coaches, Viewer players), the real design-token table, established interaction patterns, known friction points, and open questions. Read this before any UX/design work. It also holds ready-to-use Claude Design prompts.
- **`design/screenshots/`** — current-state captures referenced by the standards doc (designer position/throw/branching modes; viewer desktop/tablet/mobile).
- **`superpowers/specs/`** and **`superpowers/plans/`** — a dated spec→plan trail, one pair per shipped feature (dark HUD restyle, play designer, designer branching, drafts, mark-throw, preview, mobile drawer, position names, embedded play picker, …). This is the closest thing to a changelog of *intent*: consult the matching spec/plan when touching a feature to see the original reasoning and constraints.

When shipping a substantial feature, follow the existing convention: add a dated spec + plan here so the trail stays complete.
